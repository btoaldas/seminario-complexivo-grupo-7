🎯 Definición del Problema

⚠️ Problemática Principal

Los clubes de fútbol y agencias de scouting necesitan identificar talento potencialmente infravalorado en un mercado saturado de jugadores profesionales. El proceso manual de evaluación es costoso, lento y subjetivo.

💡 Solución Propuesta

Desarrollar un sistema inteligente de scouting que utilice Machine Learning (Regresión) para predecir el valor de mercado real de un jugador basándose en sus atributos técnicos y físicos, permitiendo identificar oportunidades donde el valor predicho supere al valor actual.

🎪 Variable Objetivo

valor_mercado_eur

Predecir el valor de mercado en euros de un jugador basándose en más de 100 atributos técnicos, físicos y características del jugador.

🔬 Metodología

Usamos una metodología tipo CRISP-DM, avanzando paso a paso desde la comprensión del negocio hasta el despliegue del sistema de analítica, organizada en 6 fases.

1

Fase 1 – Pipeline de Limpieza de Datos

Aquí programamos el proceso que transforma el archivo original en fifa_limpio.csv. Esta fase la voy a explicar en profundidad más adelante.

2

Fase 2 – EDA: Análisis Exploratorio de Datos

Ya con los datos limpios, analizamos distribuciones, patrones, outliers, correlaciones y características que influyen en el valor del jugador.

3

Fase 3 – Entrenamiento de Modelos

Entrenamos modelos de regresión. La Regresión Lineal fue nuestro baseline, pero el Random Forest alcanzó mejor desempeño para predecir el valor del jugador.

4

Fase 4 – API REST con FastAPI

Convertimos el modelo en un servicio accesible mediante HTTP: la API permite buscar jugadores, obtener estadísticas y realizar predicciones.

5

Fase 5 – Dashboard con Streamlit

Creamos una interfaz interactiva donde un usuario puede explorar jugadores, analizar el mercado y obtener predicciones sin programar.

6

Fase 6 – Docker y despliegue

Empaquetamos backend y frontend como contenedores. Con docker-compose orquestamos ambos servicios y montamos el volumen de datos.

Estas seis fases garantizan un producto final integral.

🔧 Pipeline: Limpieza de Datos

Proceso automatizado que transforma el dataset original de FIFA en un archivo limpio y listo para análisis y modelado

1

Carga del dataset original (Excel con 7 hojas)

Leemos el archivo fifa.xlsx, que trae una hoja por cada versión del juego. El pipeline recorre las siete, las carga y las une en un solo DataFrame con más de 180 mil registros brutos.

2

Unificación y renombrado de columnas

El dataset tiene 106 columnas con nombres distintos según la versión. Aquí las traducimos, normalizamos con snake_case y filtramos solo las útiles. El resultado: 73 columnas limpias y consistentes.

3

Limpieza general (duplicados, nulos y formatos)

Eliminamos duplicados, descartamos columnas con más del 70% de nulos, convertimos valores monetarios como '5.5M' o '350K' a euros numéricos, y estandarizamos fechas. Con esto, la estructura queda ordenada.

4

Imputación inteligente según posición del jugador

Los valores faltantes se rellenan considerando el rol del jugador. Un arquero nunca recibe valores de un extremo, y viceversa. Así evitamos distorsiones y respetamos el perfil real del jugador.

5

Ingeniería de características (nuevas variables)

Creamos nuevas features que añaden contexto y mejoran el modelo

Estas características enriquecen el análisis y capturan información que el dataset original no trae explícita.

🔍 EDA: Análisis Exploratorio de Datos

🎯 ¿Por Qué EDA Antes de Machine Learning?

Para aplicar modelos predictivos mediante machine learning, realizamos el Análisis Exploratorio de Datos (EDA). Esta es una fase fundamental en todo proyecto de ciencia de datos y machine learning. Consiste en explorar, comprender y resumir los datos antes de aplicar modelos predictivos o algoritmos, con el objetivo de descubrir patrones, detectar errores, evaluar la calidad de los datos y entender su comportamiento.

🤖 Machine Learning: Evaluación de Modelos de Regresión

📊 Aplicación de Machine Learning

Una vez hecho nuestro Análisis Exploratorio de Datos (EDA), aplicamos varias técnicas de machine learning enfocadas en la predicción del valor de mercado de jugadores de fútbol. Lo cual hicimos una evaluación de modelos de regresión para determinar el mejor enfoque predictivo.

🎯 ¿Por Qué Modelos de Regresión?

Los modelos de regresión se utilizan para predecir o estimar un valor numérico a partir de uno o varios datos de entrada.

Son fundamentales en análisis estadístico y en machine learning porque permiten:

✓ Entender relaciones entre variables
✓ Realizar predicciones basadas en patrones históricos
✓ Estimar valores futuros con base en datos conocidos

🔬 Proceso de Modelado Predictivo

En nuestro proceso de modelado predictivo del valor de mercado de los jugadores, se evaluaron dos enfoques principales:

1. Modelo de Regresión Lineal
2. Modelo Random Forest (ensamble de árboles de decisión)

Cada modelo fue entrenado con los mismos datos ya limpios y luego evaluamos con métricas de rendimiento para medir su capacidad de predicción.

📉 Regresión Lineal

Rol: Modelo de referencia (baseline)

Concepto: Asume una relación lineal entre las variables independientes (salario, reputación, potencial) y el valor de mercado del jugador.

✅ Ventajas:
• Fácil de interpretar
• Rápido de entrenar (<1 min)

❌ Limitaciones:
• No captura relaciones no lineales

🌲 Random Forest

Rol: Modelo de Ensamble (ganador)

Concepto: Se compone de muchos árboles de decisión que votan o promedian sus resultados.

✅ Ventajas:
• Captura relaciones complejas y no lineales
• Maneja interacciones entre variables
• Ideal para datos reales como el mercado futbolístico

❓ ¿Qué es una Relación No Lineal?

Una regresión no lineal es un tipo de modelo donde la relación entre las variables no sigue una línea recta.

Es decir, los datos no pueden explicarse con una fórmula simple y presentan comportamientos como:

📈 Curvas crecientes o decrecientes
🔝 Saturación o límite (por ejemplo, llega a un punto donde ya no mejora)
🔄 Interacciones complejas entre múltiples variables

Random Forest puede capturar estos patrones complejos

🏆 ¿Por qué elegimos Random Forest?

✔ Obtuvo mejores resultados

Los modelos de regresión lineal no pudieron capturar patrones complejos. Random Forest demostró mayor precisión en todas las métricas de evaluación.

✔ Captura patrones reales

El mercado de fichajes no es lineal. Random Forest puede identificar relaciones no lineales entre las características de los jugadores y su valor de mercado.

✔ Reduce el sobreajuste

Los modelos ensemble como Random Forest combinan múltiples árboles de decisión. Esto reduce errores y mejora la generalización en datos nuevos.

✔ Maneja interacciones complejas

Identifica automáticamente cómo se combinan múltiples variables. Por ejemplo: edad + posición + estadísticas afectan de manera conjunta el valor.

✔ Robusto ante outliers

A diferencia de la regresión lineal, Random Forest es más resistente a valores atípicos y no se ve afectado fácilmente por datos extremos.

🎯 Conclusión

El modelo Random Forest nos demostró ser más eficiente, robusto y realista para predecir el valor de mercado de los jugadores de FIFA.

Su capacidad para capturar patrones complejos y reducir el sobreajuste lo convierte en la mejor opción para nuestro sistema de valoración.

🔌 API REST: FastAPI para Scouting Inteligente

🎯 Integración del Modelo Predictivo en Producción

Para poder integrar el modelo predictivo y con el fin de poner en producción el modelo de machine learning que estima el valor de mercado de los futbolistas a partir de sus atributos técnicos, físicos y económicos, se desarrolló una API REST utilizando el framework FastAPI en Python.

FastAPI permite exponer el modelo como un servicio web ligero y de alto rendimiento para utilizar en un dashboard web que puede acceder mediante peticiones HTTP.

De esta manera, el modelo deja de ser un experimento aislado en un notebook y se convierte en un componente reutilizable dentro de una arquitectura de software.

📡 Endpoints Principales

🔍 /players/search

Función: Permite filtrar jugadores por posición, edad y potencial.

Ejemplo:
• Buscar delanteros entre 20-25 años
• Filtrar por potencial mayor a 85
• Ordenar por valor de mercado

👤 /players/{player_id}/profile

Función: Devuelve el perfil completo de un jugador, incluyendo su valor de mercado predicho por el modelo.

Ejemplo:
• Ver estadísticas completas de Messi
• Comparar valor real vs predicho

📄 Documentación Automática

Además, FastAPI genera automáticamente una documentación interactiva (Swagger/OpenAPI), lo que facilita la prueba y el consumo del servicio por parte de otros desarrolladores.

Puedes acceder a la documentación en:
• 📘 http://grupo7fifa.uniandes.site:8000/docs → Interfaz Swagger interactiva

✨ Ventajas de la API REST en FastAPI

🧠 Desacoplamiento del modelo

El modelo ML funciona de forma independiente del dashboard. Esto permite que múltiples aplicaciones (web, móvil, etc.) puedan consumir las predicciones sin duplicar código.

🌐 Acceso remoto en tiempo real

Las predicciones están disponibles mediante peticiones HTTP, lo que permite consultar el valor de mercado de cualquier jugador desde cualquier lugar con conexión a internet.

📄 Documentación automática

FastAPI genera automáticamente la documentación de los endpoints y de los esquemas de entrada/salida, facilitando la integración para otros desarrolladores.

🎯 Conclusión

Gracias a esta API, el modelo de predicción de valor de mercado puede integrarse de forma sencilla en distintos sistemas de análisis y toma de decisiones dentro del contexto del fútbol profesional.

🔐 Configuración Nginx (Reverse Proxy + SSL)

¿Por qué Nginx?
Docker expone puertos 8000 y 8501, pero usuarios finales esperan acceso por HTTPS en puerto 443 (sin :8501 en la URL). Nginx actúa como reverse proxy que:
• Recibe requests HTTPS en puerto 443
• Redirecciona internamente a localhost:8501 (Streamlit)
• Agrega certificado SSL (Let's Encrypt) para conexión segura
• Maneja compresión gzip, cache de estáticos, rate limiting

Configuración Nginx:
# /etc/nginx/sites-available/fifa-scouting
server {
  listen 80;
  server_name grupo7fifa.uniandes.site;
  return 301 https://$host$request_uri; # Redirect HTTP → HTTPS
}

server {
  listen 443 ssl;
  server_name grupo7fifa.uniandes.site;

  ssl_certificate /etc/letsencrypt/live/grupo7fifa.uniandes.site/fullchain.pem;
  ssl_certificate_key /etc/letsencrypt/live/grupo7fifa.uniandes.site/privkey.pem;

  location / {
    proxy_pass http://localhost:8501;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_set_header Host $host;
  }
}
Certificado SSL (Let's Encrypt):
certbot --nginx -d grupo7fifa.uniandes.site
Automático: Certbot modifica Nginx, agrega SSL, configura auto-renovación cada 90 días.

Tipo: bridge (red virtual privada entre contenedores)
Nombre: fifa-network
Miembros: fifa-backend + fifa-frontend

Comunicación interna:
• Frontend hace request a http://backend:8000/jugadores/buscar
• Docker resuelve backend → IP interna del contenedor (ej: 172.18.0.2)

• Frontend renderiza tabla en Streamlit

Ventajas red bridge:
✓ Contenedores se comunican por nombre (no IPs dinámicas)
✓ Aislamiento: Otros contenedores del host no pueden acceder
✓ Performance: Comunicación interna no pasa por interfaz pública (latencia ~0.1ms)
✓ Seguridad: Backend no expone puerto al exterior (solo frontend visible)

🎯 Principios de Despliegue Elegidos

1. Lo Común es Mejor para Producción

Docker + VPS + Nginx es el stack más usado en la industria. No por moda, sino porque es battle-tested con millones de despliegues exitosos. Soluciones "modernas" como serverless (AWS Lambda, Vercel) tienen limitaciones ocultas (cold starts, timeouts, vendor lock-in).

2. Control Total del Ambiente

VPS con root access significa control absoluto: instalar lo que sea (Redis, PostgreSQL, cron jobs), ajustar límites de sistema (ulimit, RAM), monitorear recursos (htop, docker stats). Plataformas PaaS (Heroku, Render) ocultan la infraestructura → Dificulta debugging.

3. Costos Predecibles Sin Sorpresas

Contabo cobra €8.99/mes fijo. AWS/GCP cobran por CPU time, requests, bandwidth → Factura variable (spike de tráfico = $500 inesperados). Para proyectos estudiantiles/académicos, presupuesto fijo es crítico.

4. Versatilidad y Escalabilidad

Docker Compose permite agregar servicios fácilmente: Redis para cache, PostgreSQL para usuarios, Celery para tareas asíncronas. Todo en docker-compose.yml, sin configurar múltiples servicios cloud.

5. Despliegue Rápido Sin Complicarse

git pull + docker-compose up -d --build → Sistema actualizado en 2 minutos. Sin CI/CD complejo (GitHub Actions, Jenkins), sin pipelines de 10 pasos. Para MVP y prototipos, simplicidad > sofisticación.
